## Root Cause Analysis

The test is attempting to cast the WebDriver instance to an incorrect type. The WebDriver factory method returns an `EventFiringWebDriver` (a wrapper/decorator around the base driver used for event handling), but the test code performs a direct cast to `OpenQA.Selenium.WebDriver`. Since `EventFiringWebDriver` does not inherit from `WebDriver` and only implements the `IWebDriver` interface, the explicit cast fails at runtime with a `System.InvalidCastException`.

## Evidence

```

'OpenQA.Selenium.Support.Events.EventFiringWebDriver'
to type 'OpenQA.Selenium.WebDriver'.
```

The error occurs consistently for both Firefox and Chrome browsers, confirming this is a code-level issue rather than a browser-specific or environment-specific problem.

A secondary error was also observed:

This is likely a downstream failure caused by the test proceeding with a null or invalid driver reference after the cast exception is swallowed or handled incorrectly.

## Suggested Fix

1. **QA / Dev:** Update the WebDriver factory method (or test setup code) to avoid casting the returned driver to the concrete `OpenQA.Selenium.WebDriver` type. Instead, use the `IWebDriver` interface throughout the test and page-object code so that both plain and event-firing drivers are accepted without an explicit cast.

   ```csharp
   // Before (throws InvalidCastException):
   var driver = (OpenQA.Selenium.WebDriver)driverFactory.Create();

   // After (uses the common interface):
   IWebDriver driver = driverFactory.Create();
   ```

2. **Dev:** If the concrete `WebDriver` type is needed to access members not exposed by `IWebDriver` (e.g. `ExecuteScript`), unwrap the underlying driver from `EventFiringWebDriver` using its `WrappedDriver` property:

   ```csharp
   IWebDriver raw = driver is EventFiringWebDriver efwd
       ? efwd.WrappedDriver
       : driver;
   var jsExecutor = (IJavaScriptExecutor)raw;
   ```

3. **QA:** Retest `AddingAShiftAndSavingToUnpostedWeekWillNotGenerateOfferCS` on both Firefox and Chrome after applying the fix to confirm the `InvalidCastException` and the secondary `DeSelect` failure are resolved.

## Copilot Investigation Notes

### Problem

The `EventFiringWebDriver` class in Selenium is a decorator that wraps an `IWebDriver` implementation and adds event-raising hooks (e.g., `FindingElement`, `ElementClicked`). It implements `IWebDriver` but **does not extend** the abstract `WebDriver` base class. Therefore, casting an `EventFiringWebDriver` instance to `WebDriver` will always throw `System.InvalidCastException` at runtime, regardless of the browser being used.

### Why it is persistent

The WebDriver factory used in the test suite consistently returns `EventFiringWebDriver` (to enable event logging or screenshots on failure). As long as any test code contains a hard cast to `WebDriver`, it will always fail, explaining the persistent pattern observed since 2025-09-29.

### Fix summary

- Replace all hard casts `(WebDriver)driver` with the `IWebDriver` interface type.
- Where a concrete driver capability is required (e.g., `IJavaScriptExecutor`, `ITakesScreenshot`), cast to the appropriate interface or unwrap via `EventFiringWebDriver.WrappedDriver`.
- No changes to the factory method or event-listener configuration are necessary.
